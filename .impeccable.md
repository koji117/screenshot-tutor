## Design Context

### Users
A solo developer studying for certifications. Primary context is **macOS at a desk** during long reading sessions, with a growing **iPad** secondary context — sometimes handheld, sometimes paired with a Bluetooth keyboard, often in different lighting than the desk. Often pastes textbook pages, slide decks, code snippets, and exam problems. Bilingual: reads English and Japanese.

### Brand Personality
**Calm. Scholarly. Patient.**

This is a quiet study companion, not an AI tool that demands attention. It treats the user's time as serious — long reading sessions, focused work. Warm but disciplined. The screenshot is the primary citizen.

This personality is the same across web and iOS — the *expression* of it differs (see below).

### Aesthetic Directions

The product ships on two surfaces with two distinct visual languages. Same brand voice, different idioms.

#### Web app (`/` — Cloudflare Workers)
**Refined editorial — a quiet study desk in a library.**

- Light mode default (long reading sessions favor lighter backgrounds; reduces eye strain in lit rooms).
- Generous whitespace, careful spacing rhythm.
- Distinctive typography: a warm editorial serif paired with a clean geometric sans. Both must work in EN and JA.
- Restrained color: warm off-white background, dark ink foreground, one deep ink-blue accent. Tinted neutrals (no pure white/black).
- No gradients, no glassmorphism, no glowing cards.

**References:** Are.na, Read.cv, classic literary journals. Chat bubbles feel like marginal notes in a textbook, not Slack messages.

**Anti-references:** Generic AI app aesthetics (cyan-on-dark, neon accents, gradient text). Default Inter/system-stack typography. ChatGPT-style chat UI. Templated dashboard card grids.

#### iOS native app (`/ios` — `native-ipad-app` branch)
**Native iOS — feels like an Apple-platform app, not a port.**

- Use SF Pro / SF Pro Rounded as the system intends; no custom font cargo-culting from the web app.
- System tints, system-blue buttons, system-grouped backgrounds, native `Form` and `List` styles.
- Honor light *and* dark mode via the system appearance — no custom theming layer.
- Standard SwiftUI affordances: `Menu` for picker overflow, `ShareLink` for export, `NavigationStack` for routing, `fullScreenCover` for transient overlays.
- Embrace iOS conventions iPad users already know — adopting platform language is a sign of respect, not a lack of design opinion.

The editorial aesthetic from the web app does **not** carry over. The two apps share data formats (markdown export, screenshot frontmatter) and feature naming, not visuals.

### Design Principles
1. **Reading is the primary act.** The summary / breakdown / synthesis is the focus; supporting UI recedes once the user is reading.
2. **Quiet, but practical.** Restrained palette, restrained typography. The model panel and load button take real estate because model loading is a real cost the user has to understand — that's earned chrome, not decoration.
3. **Tool, not toy.** Functional and dignified. No gamified affordances or playful touches.
4. **Bilingual-aware.** Typography pair (web) and string layout (both) must work for Latin and CJK with consistent rhythm.
5. **Respect the screenshot.** The user's pasted image is a primary citizen, not a thumbnail. Inline embeds in exports, full-bleed previews in session views.
6. **Match the platform.** The web app commits to its editorial direction; the iOS app commits to native iOS conventions. Don't force one onto the other.
