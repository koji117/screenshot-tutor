# CLAUDE.md

Repository context for AI assistants working on this codebase.

## What this repo is

**Screenshot Tutor** — a study companion that takes a screenshot
(textbook page, slide, code snippet, exam problem) and produces a
study-friendly summary, optional study breakdown, and follow-up
chat about it. Output supports English and Japanese. All inference
happens on-device — no API keys, no backend, no network round-trips
for generation.

The repo houses **two separate implementations** of the same
product, each shipping independently on its own branch:

| Branch | Implementation | Lives in | Deploy target |
|--------|----------------|----------|---------------|
| `main` | Browser / PWA — Transformers.js + WebGPU | repo root | Cloudflare Workers ([live URL](https://screenshot-tutor.koji11738.workers.dev/)) |
| `native-ipad-app` | iPad / iOS native — MLX-Swift | `ios/` subdirectory | Sideload via Xcode (paid Apple Developer team) |

Both apps share design context (`.impeccable.md`) and an Obsidian
export format (markdown + `attachments/` subfolder), but the code
is independent — no shared modules, no monorepo tooling.

## When to use which branch

- **Touching the web app?** Work on `main`. Code lives at the repo
  root: `index.html`, `js/`, `css/`, `src/index.js` (the Cloudflare
  Worker that proxies HF model fetches).
- **Touching the iOS app?** Work on `native-ipad-app`. Code lives
  in `ios/ScreenshotTutor/`. The iOS branch *also* contains the
  web app (it diverged from `main`), but don't edit the web code
  there — keep web changes on `main` and let `native-ipad-app`
  rebase forward periodically.
- **Updating shared docs** (this file, root `README.md`,
  `.impeccable.md`): land on whichever branch is active, then
  cherry-pick to the other if relevant.

## Architecture

### Web app (`main`)

- **Static-only**: no build step, no npm. Pure ES modules served
  by a Cloudflare Worker (`src/index.js`) which also proxies
  `/hf/*` to `huggingface.co` so model downloads are same-origin.
- **Worker thread** (`js/worker.js`) hosts the LLM via
  `@huggingface/transformers@4.2.0` on WebGPU. Models are
  registered in `js/models.js` (id → repo + family). Adding a
  model family means adding an entry to `FAMILIES` in `worker.js`
  with its `load`, `eosTokenStr`, and `manualPrompt`.
- **State**: `localStorage` for sessions and settings. Auto-trims
  under 4.5MB.
- **Models** (4 entries): `smolvlm-256m`, `smolvlm-500m`,
  `gemma4-e2b`, `gemma4-e4b`.
- **iOS Safari notes**: Gemma 4 models often exceed Safari's
  per-tab memory budget; the SmolVLM models are the iPad-friendly
  fallback. The native iOS app exists specifically to escape that
  budget — see below.

### iOS native app (`native-ipad-app`, in `ios/`)

- **SwiftUI** targeting iOS / iPadOS 17+, generated by
  **XcodeGen** from `ios/project.yml` (Xcode project file is
  gitignored — regenerate with `xcodegen generate`).
- **Inference**: [`mlx-swift-lm`](https://github.com/ml-explore/mlx-swift-lm)
  (Apple's MLX-based LLM/VLM library, split out of
  `mlx-swift-examples` in late 2025). Provides `MLXVLM`,
  `MLXLMCommon`. We deliberately avoid `MLXHuggingFace` because
  its macros require Xcode trust steps that produce confusing
  "Missing package product" errors when skipped — instead we
  inline the downloader/tokenizer-loader bridges in
  `HuggingFaceBridge.swift`.
- **Models** (2 entries — Gemma 4 only): E2B (~1.5GB) and E4B
  (~3GB). E4B is the default; requires the increased-memory and
  extended-virtual-addressing entitlements in
  `ScreenshotTutor.entitlements`, which require a **paid Apple
  Developer Program team** to sign. The team ID is wired into
  `project.yml` (`DEVELOPMENT_TEAM`).
- **Layout**: `NavigationSplitView` — history sidebar +
  detail. On compact widths (iPhone, narrow Slide Over) it
  collapses to a single column with the sidebar accessible via
  the standard system toggle.
- **Real device required**: MLX's Metal kernels do not run in
  the iOS Simulator. The simulator is fine for SwiftUI smoke-
  tests but can't load or run a model.

## Testing

- **Web**: `python3 -m http.server 8000` from the repo root, open
  `http://localhost:8000`. Per-component unit tests at
  `js/store.test.html`, `js/input.test.html`, `js/i18n.test.html`.
  See `TESTING.md` for the manual acceptance checklist.
- **iOS**: build & run on device via Xcode (regenerate the
  project from `ios/project.yml` first if it doesn't exist). No
  XCTest suite yet.

## Conventions

- **Commits**: Conventional Commits style (`feat(scope):`,
  `fix(scope):`, `chore(scope):`, `perf(scope):`, `polish(ios):`,
  `revert(ios):`, etc.). Web changes use `feat(web):` /
  `fix(web):`. iOS changes use `feat(ios):` / `fix(ios):`. Body
  describes the *why*, not the *what*.
- **Co-authored-by**: All Claude-assisted commits include the
  trailer `Co-Authored-By: Claude Opus 4.7 (1M context)
  <noreply@anthropic.com>`.
- **Branches**:
  - `main` — web app. Auto-deploys to the live Cloudflare Worker
    on push.
  - `native-ipad-app` — iOS app. No CI; the developer pulls the
    branch on their Mac and builds locally.
- **Design context**: `.impeccable.md` at the repo root. Two
  aesthetic directions — web is "refined editorial" (Spectral +
  Manrope, ink-blue accent, light mode); iOS is "native iOS"
  (SF Pro, system tints, light *and* dark mode via system
  appearance). Don't bleed one into the other.

## Common gotchas

- **iOS Simulator + MLX**: don't try. The runtime detects the
  simulator and fails fast with a clear message; running on
  device is the only option.
- **Xcode "Missing package product" errors**: regenerate the
  project (`rm -rf ScreenshotTutor.xcodeproj && xcodegen
  generate`). The package graph is in `project.yml`, not in the
  pbxproj — the latter is gitignored.
- **iOS paste prompts**: the input row uses system `PasteButton`
  controls, which iOS treats as user-authorized and skips the
  "Allow Paste" prompt. Don't replace them with regular Buttons
  reading `UIPasteboard.general` — that brings the prompt back.
- **Web Worker cache**: bumping `WORKER_VERSION` in `js/app.js`
  forces browsers to re-fetch the worker script after a deploy
  (the worker is cached separately from the page).
- **Markdown export structure**: the `attachments/` folder must
  be a sibling of the `.md` file when imported into Obsidian, or
  the inline `![[attachments/...]]` wikilinks won't resolve. iOS
  uses `UIDocumentPicker(forExporting:asCopy:)` (preserves
  folders); the web app uses File System Access API which
  creates the subfolder explicitly.

## Skill / command notes for AI assistants

- The user runs `/audit`, `/polish`, `/optimize`, `/critique` from
  the impeccable suite. Each has its own preparation step that
  invokes `/impeccable` for design context.
- The user has `/smart-commit` available — group changes into
  logical commits by concern, present a plan, then execute.
- Push policy: `git push` is a confirm-required action per
  `~/.claude/CLAUDE.md` (user's global). Always confirm before
  pushing unless the user explicitly authorized in the same
  message.
